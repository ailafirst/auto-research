下面是一份可直接作为立项/开发用的 **Deep Research Agent 项目需求文档 PRD**。我按“学习项目 + 可扩展工程项目”的标准来写，技术栈包含 **LangGraph、RAG、FastAPI、向量数据库、LLM、多轮检索与事实核查**。

# Deep Research Agent 项目需求文档

## 1. 项目名称

Deep Research Agent：基于 LangGraph、RAG 与 FastAPI 的自动化深度调研系统

---

## 2. 项目背景

在日常学习、科研、产品分析、行业研究和竞品调研场景中，用户通常需要花费大量时间完成资料搜索、信息筛选、内容阅读、事实核查和报告撰写。传统搜索引擎只能返回网页链接，普通大模型虽然可以生成总结，但存在信息来源不透明、时效性不足、事实幻觉、缺乏可追溯引用等问题。

本项目希望构建一个 Deep Research Agent 系统，使用户只需输入一个研究问题，系统即可自动完成：

1. 研究问题拆解；
2. 多轮联网检索；
3. 网页内容抓取与清洗；
4. 信源质量评估；
5. 文档切片与向量化入库；
6. 基于 RAG 的证据检索；
7. 多轮分析与事实核查；
8. 生成结构化研究报告；
9. 通过 API 或 Web 页面对外提供服务。

该项目既可作为 AI Agent 工程化学习项目，也可作为后续扩展为知识调研平台、企业内部研究助手或自动化分析工具的基础。

---

## 3. 项目目标

### 3.1 核心目标

构建一个能够自动完成深度研究任务的智能 Agent 系统，支持用户输入自然语言问题，系统输出带有引用来源、结构清晰、逻辑完整的研究报告。

### 3.2 学习目标

通过本项目掌握以下技术能力：

1. 使用 LangGraph 构建多节点、多状态、多轮循环的 Agent 工作流；
2. 使用 RAG 构建基于证据的问答和报告生成流程；
3. 使用向量数据库存储和检索网页内容；
4. 使用 FastAPI 对 Agent 能力进行服务化封装；
5. 使用异步任务、日志、状态跟踪和错误处理提升工程完整度；
6. 掌握 Deep Research 类系统中的规划、检索、分析、核查和报告生成链路。

### 3.3 产品目标

系统应支持以下能力：

1. 用户输入一个研究主题或复杂问题；
2. 系统自动拆解为多个子问题；
3. 系统自动调用搜索工具获取网页资料；
4. 系统对网页内容进行清洗、筛选、切片和向量化；
5. 系统基于检索证据生成阶段性分析；
6. 系统对分析结果进行事实核查；
7. 若证据不足，系统自动进行补充检索；
8. 系统最终生成结构化研究报告；
9. 用户可以通过 REST API 或简单 Web UI 使用系统。

---

## 4. 用户角色

### 4.1 普通用户

普通用户通过 Web 页面或 API 输入研究问题，查看系统生成的调研报告。

典型需求：

1. 输入问题；
2. 查看研究进度；
3. 获取最终报告；
4. 查看引用来源；
5. 下载或复制报告。

### 4.2 开发者用户

开发者用户关注系统的内部流程、日志、状态和扩展能力。

典型需求：

1. 查看 Agent 执行链路；
2. 调试 LangGraph 节点；
3. 更换 LLM 模型；
4. 调整检索轮数和 RAG 参数；
5. 扩展搜索工具、数据库或报告模板。

### 4.3 管理员用户，后续扩展

管理员用户用于管理系统配置、任务记录、用户权限和运行监控。

初期可不实现完整管理员后台，但应预留扩展接口。

## 6. 功能需求

## 6.1 用户输入模块

### 功能描述

用户可以提交一个自然语言研究问题，并配置部分可选参数。

### 输入字段

| 字段                | 类型      | 必填 | 说明                             |
| ----------------- | ------- | -- | ------------------------------ |
| query             | string  | 是  | 用户输入的研究问题                      |
| max_rounds        | integer | 否  | 最大研究轮数，默认 2                    |
| language          | string  | 否  | 输出语言，默认 zh-CN                  |
| report_type       | string  | 否  | 报告类型，如 summary、deep、comparison |
| search_depth      | string  | 否  | 搜索深度，如 basic、advanced          |
| top_k             | integer | 否  | RAG 检索片段数量                     |
| enable_fact_check | boolean | 否  | 是否启用事实核查                       |

### 验收标准

1. 用户可以提交非空研究问题；
2. 系统能够校验输入长度；
3. 系统能够返回任务 ID；
4. 系统能够根据参数调整研究流程。

---

## 6.2 研究规划模块 Planner

### 功能描述

Planner 负责将用户问题拆解为可执行的研究计划。

### 主要能力

1. 识别用户研究意图；
2. 判断问题类型，例如解释类、对比类、趋势类、方案类；
3. 拆解 3 到 6 个子问题；
4. 为每个子问题生成搜索关键词；
5. 生成研究计划结构。

### 输出结构示例

```json
{
  "research_goal": "分析 AI Agent 行业发展趋势",
  "question_type": "trend_analysis",
  "sub_questions": [
    {
      "id": "q1",
      "question": "AI Agent 当前的主要技术路线是什么？",
      "search_queries": [
        "AI Agent architecture trend 2026",
        "LLM agent framework comparison"
      ]
    }
  ]
}
```

### 技术实现

使用 LangGraph 节点 `planner_node` 调用 LLM 生成结构化 JSON 计划。

### 验收标准

1. 能够生成结构化研究计划；
2. 子问题与原始问题相关；
3. 搜索关键词可用于联网检索；
4. 对比类问题应保留所有对比对象。

---

## 6.3 信息检索模块 Retriever

### 功能描述

Retriever 根据研究计划自动调用搜索工具，获取相关网页链接和基础摘要。

### 主要能力

1. 根据子问题执行搜索；
2. 支持多关键词检索；
3. 支持搜索结果去重；
4. 支持限制每个问题的搜索数量；
5. 支持失败重试；
6. 支持记录检索来源。

### 可选搜索工具

1. Tavily Search API；
2. Serper API；
3. Bing Search API；
4. 自定义搜索接口。

### 输出字段

| 字段              | 说明     |
| --------------- | ------ |
| title           | 页面标题   |
| url             | 页面 URL |
| snippet         | 搜索摘要   |
| source          | 搜索来源   |
| sub_question_id | 关联子问题  |
| retrieved_at    | 检索时间   |

### 验收标准

1. 每个子问题至少获取若干候选来源；
2. URL 不应大量重复；
3. 搜索失败时不会导致整个任务崩溃；
4. 检索结果应写入任务状态。

---

## 6.4 网页抓取与内容清洗模块

### 功能描述

系统根据搜索结果抓取网页正文，并对网页内容进行清洗。

### 主要能力

1. 抓取网页正文；
2. 移除导航栏、广告、版权区、无效文本；
3. 过滤内容过短或质量较低的页面；
4. 保留标题、URL、发布时间、作者等元数据；
5. 支持抓取失败重试；
6. 支持超时控制。

### 可选实现方式

1. Jina Reader；
2. trafilatura；
3. newspaper3k；
4. BeautifulSoup + readability；
5. Playwright，作为复杂动态页面的后续扩展。

### 验收标准

1. 能够从常见网页中提取正文；
2. 正文长度低于阈值的页面会被过滤；
3. 保留 URL、标题等来源信息；
4. 抓取失败有日志记录。

---

## 6.5 信源评估模块 Source Evaluator

### 功能描述

Source Evaluator 对搜索结果和网页正文进行质量评估，筛选可用于分析的资料。

### 评估维度

1. 内容相关性；
2. 正文长度；
3. 域名可信度；
4. 页面是否过时；
5. 是否来自低质量聚合站；
6. 是否与用户问题直接相关；
7. 是否包含可验证信息。

### 评分结构示例

```json
{
  "url": "https://example.com/article",
  "relevance_score": 0.82,
  "credibility_score": 0.75,
  "freshness_score": 0.68,
  "final_score": 0.76,
  "accepted": true,
  "reason": "内容与研究问题高度相关，来源质量较好"
}
```

### 验收标准

1. 系统可以过滤明显无关网页；
2. 系统可以保留高质量信源；
3. 每个被保留来源应有评分或筛选理由；
4. 信源评估结果应进入后续 RAG 流程。

---

## 6.6 RAG 知识构建模块 Evidence Builder

### 功能描述

Evidence Builder 负责将网页正文转换为可检索证据，包括切片、向量化和入库。

### 主要能力

1. 文档切片；
2. 文本清洗；
3. Embedding 生成；
4. 向量写入数据库；
5. 元数据绑定；
6. 按任务 ID 隔离数据；
7. 支持后续证据检索。

### 推荐技术栈

| 能力    | 技术                                    |
| ----- | ------------------------------------- |
| 文本切片  | 自定义 chunker 或 LangChain text splitter |
| 向量模型  | fastembed、bge、OpenAI embedding        |
| 向量数据库 | Qdrant                                |
| 元数据管理 | PostgreSQL 或 SQLite                   |
| 检索接口  | qdrant-client                         |

### Chunk 元数据

```json
{
  "task_id": "task_001",
  "source_id": "source_001",
  "url": "https://example.com",
  "title": "Example Article",
  "chunk_index": 3,
  "text": "chunk text...",
  "created_at": "2026-06-17T10:00:00Z"
}
```

### 验收标准

1. 网页正文可以被成功切片；
2. 每个 chunk 可以生成 embedding；
3. chunk 可以写入 Qdrant；
4. 检索时可以按 task_id 过滤；
5. 检索结果能返回原始 URL 和标题。

---

## 6.7 分析模块 Analyst

### 功能描述

Analyst 基于 RAG 检索到的证据片段，对每个子问题生成分析内容。

### 主要能力

1. 按子问题检索相关证据；
2. 基于证据生成回答；
3. 保留引用来源编号；
4. 避免无证据推断；
5. 标记证据不足的问题。

### 输出示例

```json
{
  "sub_question_id": "q1",
  "answer": "AI Agent 当前主要技术路线包括工作流型 Agent、多工具调用 Agent 和多智能体协作系统...",
  "citations": ["S1", "S3", "S7"],
  "confidence": 0.78,
  "evidence_gap": false
}
```

### 验收标准

1. 每个子问题都有对应分析结果；
2. 分析内容应引用证据；
3. 分析不得明显脱离检索材料；
4. 证据不足时应显式标记。

---

## 6.8 事实核查模块 Fact Checker

### 功能描述

Fact Checker 对 Analyst 生成的分析内容进行核查，判断是否存在事实错误、证据不足或逻辑跳跃。

### 主要能力

1. 检查关键结论是否有证据支持；
2. 检查引用是否对应原文；
3. 检查不同来源之间是否存在冲突；
4. 检查是否出现过度推断；
5. 生成补充检索建议；
6. 决定是否进入下一轮检索。

### 输出示例

```json
{
  "passed": false,
  "issues": [
    {
      "type": "insufficient_evidence",
      "claim": "某技术将在 2026 年成为主流",
      "reason": "当前证据只显示该技术正在增长，但不足以证明将成为主流"
    }
  ],
  "follow_up_queries": [
    "2026 AI Agent adoption statistics",
    "enterprise AI agent market report 2026"
  ]
}
```

### 验收标准

1. 能够发现明显证据不足的问题；
2. 能够生成补充检索关键词；
3. 当核查失败且未超过最大轮数时，系统会自动进入下一轮检索；
4. 达到最大轮数后，系统应输出带有限制说明的报告。

---

## 6.9 报告生成模块 Report Writer

### 功能描述

Report Writer 将研究计划、分析结果、证据来源和事实核查结果整合为最终报告。

### 报告结构

最终报告应包括：

1. 标题；
2. 摘要；
3. 研究问题说明；
4. 核心结论；
5. 分章节分析；
6. 对比表或趋势表，视问题类型而定；
7. 风险与不确定性；
8. 证据不足说明；
9. 参考来源列表。

### 报告格式

支持以下格式：

1. Markdown；
2. JSON；
3. 后续可扩展为 PDF、DOCX。

### 验收标准

1. 报告结构清晰；
2. 结论有引用来源；
3. 不确定信息有明确说明；
4. 对比类问题应包含表格；
5. Markdown 格式可直接复制使用。

---

## 6.10 任务管理模块

### 功能描述

系统应支持异步任务管理，用户提交研究问题后，可以查询任务状态和结果。

### 任务状态

| 状态            | 说明           |
| ------------- | ------------ |
| pending       | 等待执行         |
| planning      | 正在生成研究计划     |
| retrieving    | 正在检索资料       |
| indexing      | 正在构建 RAG 证据库 |
| analyzing     | 正在分析         |
| fact_checking | 正在事实核查       |
| writing       | 正在生成报告       |
| completed     | 已完成          |
| failed        | 执行失败         |

### 验收标准

1. 用户提交任务后立即获得 task_id；
2. 用户可以查询任务状态；
3. 任务失败时返回错误信息；
4. 任务完成后可以获取最终报告。

## 6.11 API 服务模块

### 功能描述

使用 FastAPI 对外提供 Deep Research 服务接口。

### API 设计

#### 1. 创建研究任务

```http
POST /api/research
```

请求体：

```json
{
  "query": "对比 LangGraph 和 CrewAI 的区别",
  "max_rounds": 2,
  "language": "zh-CN",
  "report_type": "comparison"
}
```

响应体：

```json
{
  "task_id": "task_001",
  "status": "pending"
}
```

#### 2. 查询任务状态

```http
GET /api/research/{task_id}/status
```

响应体：

```json
{
  "task_id": "task_001",
  "status": "analyzing",
  "progress": 65
}
```

#### 3. 获取研究报告

```http
GET /api/research/{task_id}/report
```

响应体：

```json
{
  "task_id": "task_001",
  "status": "completed",
  "report": "# LangGraph 与 CrewAI 对比分析..."
}
```

#### 4. 获取任务详情

```http
GET /api/research/{task_id}
```

响应体包括：

1. 原始问题；
2. 研究计划；
3. 检索来源；
4. 分析结果；
5. 事实核查结果；
6. 最终报告。

### 验收标准

1. API 可以通过 Swagger 文档访问；
2. 请求参数有 Pydantic 校验；
3. 异常返回统一格式；
4. 支持跨域配置；
5. 支持基础日志记录。

## 7. 非功能需求

## 7.1 性能需求

1. 普通研究任务应在可接受时间内完成；
2. 单个任务应限制最大搜索次数、最大网页数和最大 token 使用量；
3. RAG 检索应支持 top_k 参数；
4. API 不应因单个长任务阻塞其他请求。

## 7.2 稳定性需求

1. 外部搜索 API 调用失败时应重试；
2. 网页抓取失败不应导致整个任务失败；
3. LLM 调用超时时应返回明确错误；
4. Qdrant 不可用时应有降级或错误提示；
5. 每个节点应有日志记录。

## 7.3 可扩展性需求

1. 支持替换 LLM 模型；
2. 支持替换搜索 API；
3. 支持替换向量数据库；
4. 支持增加新的 LangGraph 节点；
5. 支持增加新的报告模板；

## 7.4 安全需求

1. API Key 不得硬编码；
2. 所有密钥通过环境变量管理；
3. 用户输入需要长度限制；
4. 外部 URL 抓取需要超时限制；
5. 后续应支持用户鉴权和限流。

## 7.5 可观测性需求

1. 记录每个任务的执行状态；
2. 记录每个 LangGraph 节点耗时；
3. 记录 LLM 调用次数和 token 消耗；
4. 记录检索来源数量；
5. 记录失败原因；
6. 后续可接入 Prometheus、Grafana 或 OpenTelemetry。

## 15. 风险与应对方案

## 15.1 LLM 幻觉风险

风险：模型可能生成无来源支撑的结论。

应对：

1. 强制基于证据生成；
2. 报告中保留引用；
3. 增加事实核查节点；
4. 对证据不足内容显式标记。

## 15.2 搜索质量不稳定

风险：搜索 API 返回结果质量不稳定。

应对：

1. 多关键词搜索；
2. 信源评分；
3. URL 去重；
4. 支持更换搜索供应商。

## 15.3 网页抓取失败

风险：部分网页无法抓取正文。

应对：

1. 设置超时和重试；
2. 使用多个正文抽取方案；
3. 抓取失败时跳过该来源；
4. 保留失败日志。

## 15.4 成本过高

风险：多轮检索和多次 LLM 调用导致成本上升。

应对：

1. 限制最大轮数；
2. 限制每轮搜索数量；
3. 控制 RAG top_k；
4. 使用小模型处理部分节点；
5. 增加 token 统计。

## 15.5 响应时间过长

风险：Deep Research 任务执行时间较长。

应对：

1. 使用异步任务；
2. 前端轮询任务状态；
3. 对搜索和抓取进行并发处理；
4. 设置最大网页数和超时时间。
