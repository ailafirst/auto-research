# 版本说明 v0.1.4

> 发布日期：2026-06-25 · 上一版本：[v0.1.3](#) · 对比基线：commit `bb7cd4a`

本版本聚焦**检索质量**与**运行性能**两条主线：修复了 RAG 切片中导致"文本与向量不匹配"的核心缺陷，重构了 LLM 并发体系，并对 Tavily 搜索稳健性、启动冷启动、短输出误判等问题做了系统性改进。

## 二、RAG 检索质量（核心）

### 2.1 切片巨块按句子边界切分

**改动**：`TextChunker` 新增 `_split_oversized()`，对超过 `chunk_size` 的段落在句子边界处二次切分。

**原因**：自 v0.1.4 统一使用 Tavily `raw_content`（整页正文）后，`text.split("\n\n")` 假设 `\n\n` 标记段落边界，但 markdown 化的正文经常**丢失段落边界**——多个真实段落被单个 `\n` 连接。于是被当作"一个段落"的实际是 3000+ 字的巨块。而下游三个环节全部硬截断：

| 环节 | 限制 | 对 3860 字 chunk 的实际效果 |
|------|------|------|
| Embedding | `max_seq_length=512` token | 向量只编码前 ~1/3 |
| Rerank | `text[:512]` 字符 | 打分只看前 ~1/7 |
| 证据筛选 | `score > 0.45` | 基于模糊向量的分数，失去区分意义 |

**会导致什么**：巨块向量是"前 512 token 的模糊平均"，对任何查询都呈中等相似——既不高也不低。表现为 `score_std ≈ 0.01`（检索分数无区分度），阈值形同虚设，且单篇长文章垄断检索 top-20，压低来源多样性。

**怎么改的**：超长段落按中英文句末标点（`。！？!?；;`）和换行二次切分为 ≤`chunk_size` 的片段，**只在句子边界断开，绝不切断句子**；仅当单句本身超长（无标点长串，罕见）时才硬切兜底。正常段落原样保留语义完整性。

**实测效果**（任务 05/07/09，P90 字数）：

| 任务 | 修复前 | 修复后 | 变化 |
|------|--------|--------|------|
| 05 AI芯片 | 2845 | 977 | **-66%** |
| 07 碳中和 | 2639 | 1002 | **-62%** |
| 09 LLM幻觉 | 3272 | 1001 | **-69%** |

P90 精确收敛到 ~1000（=chunk_size 800 + overlap 200），3000+ 字巨块被彻底消除。`score_std` 同步上升 25%~65%，确认区分度提升。

### 2.2 修复中文 overlap 退化 bug

**改动**：chunk overlap 从"按词切分"改为"按字符截取"。

**原因**：旧逻辑 `current_chunk.split()` 按空格分词取 overlap，但**中文没有空格**，`.split()` 把整段当成一个词，`overlap` 退化为**整个上一个 chunk**。

**会导致什么**：每个中文 chunk 都被前一个 chunk 的全文撑大近一倍，是切片巨块化的又一来源。

**怎么改的**：`overlap_text = current_chunk[-self.overlap:]` 直接按字符截取最后 200 字，对中英文一致正确。

### 2.3 Rerank 截断上限放宽

**改动**：`rerank_chunks` 中 `text[:512]` → `text[:1024]`。

**原因**：rerank 按前 512 字符给 chunk 打分，即便正常 800 字 chunk 也漏掉后 ~288 字。

**会导致什么**：rerank 打分依据不完整，可能错排。放宽到 1024 后完整覆盖切分后的 chunk（≤chunk_size+overlap≈1000）。

> RAG 模型栈（bge-m3 embedding、bge-reranker-v2-m3 重排序）与评估指标体系详见 [`RAG改进方案.md`](RAG改进方案.md)。重排序通过 `RERANKER_ENABLED=true` 开启。

## 三、LLM 并发体系重构

**改动**：新增全局 LLM 并发信号量与 4 个可配置并发参数。

**原因**：v0.1.3 在 `llm_service.py` **没有全局并发限制**，各节点各自硬编码（analyst `Semaphore(5)`、fact_checker `Semaphore(3)`），无法统一管控，也无法为生产/基准设置不同预算。实测 mimo 后端真实瓶颈是 req/min 而非并发连接数，安全并发可达 12。

**会导致什么**：统一信号量后，全流程 LLM 调用受单一上限保护，避免触发后端限速；生产与基准互不干扰。

**怎么改的**：

| 配置项 | 默认值 | 说明 | v0.1.3 |
|--------|--------|------|--------|
| `LLM_MAX_CONCURRENCY` | 12 | 全局 LLM 并发上限（mimo 实测安全值）| 无（无全局限制）|
| `ANALYST_CONCURRENCY` | 8 | analyst 节点内子问题并发 | 硬编码 5 |
| `FACT_CHECKER_CONCURRENCY` | 5 | fact_checker 节点内子问题并发 | 硬编码 3 |
| `LLM_BENCHMARK_CONCURRENCY` | 10 | 基准总 LLM 预算（pipeline+RAGAS 共享）| 无 |

`llm_service.py` 的 `litellm.acompletion` 调用统一包裹在 `_llm_semaphore` 内；基准进程通过替换模块级信号量对象，使 pipeline 与 RAGAS 共享同一预算。

## 四、Tavily 搜索稳健性

### 4.1 key 池探针（Plan A）

**改动**：`SearchService.probe_key_pool()`，在 `retriever_node` 批量搜索前调用。

**原因**：key 池顺序轮换，当排在前面的 key 配额耗尽时，N 个并发搜索会**同时撞到同一耗尽 key**，各自串行重试+轮换，形成级联失败风暴（实测某任务 retriever 达 325s）。

**会导致什么**：探针先发 1 次轻量请求（basic、不耗 advanced credit）确认当前 key 可用，耗尽则提前轮换，使后续并发搜索直接命中可用 key。

**怎么改的**：探针在批量 `asyncio.gather` 之前串行执行，最多轮换 `len(keys)+1` 次找到可用 key；非配额错误（网络抖动）保留当前 key 交由正常重试。

### 4.2 退避封顶

**改动**：tenacity 指数退避 `max=30` → `max=8` 秒。

**原因**：限速重试单次最高等待 30s，放大了失败任务的总耗时。

**会导致什么**：失败重试的等待被压到最高 8s，收紧可控的那部分耗时。

> 注：曾评估提高 Tavily 并发（Semaphore 3→5）+ 缩短请求间隔（0.5→0.2s），但二者会提升单 key 请求速率、**更易触发 429 限速**，与稳健目标相悖，故撤销，仅保留退避封顶。retriever 在 key 中途耗尽时的慢本质是配额问题，需补充可用 key 解决。

### 4.3 raw_content 统一

**改动**：生产与基准统一使用 `advanced + raw_content=True`（默认值）。

**原因**：basic 模式只返回摘要，正文需 `content_extractor` 爬虫补抓，增加延迟与失败风险。基准曾降级 basic 以省配额。

**会导致什么**：Tavily 直接返回整页正文，`content_extractor` 耗时降为 0s（无需爬取），生产与基准行为一致。探针（4.1）消除了 raw_content 大 payload 的级联失败风险。

## 五、启动模型预热（Plan D）

**改动**：FastAPI `lifespan` 与基准脚本启动时预热 embedding / reranker 模型。

**原因**：bge-m3（~20-30s）+ bge-reranker（~5-10s）首次加载耗时被算进第一个请求的 `evidence_builder` 节点，造成首请求异常慢。

**会导致什么**：模型加载从节点耗时中剥离。实测 `evidence_builder` 从 ~45s 降至 ~20s（纯 embed 计算），`analyst` 从 ~23s 降至 ~7s。基准 12 任务共享单次模型加载（~30s 均摊到每任务仅 2.5s）。

**怎么改的**：`lifespan` 在 Qdrant 检查后预热 `get_rag_service()` 的 embedding 模型（按 provider）与 `_get_reranker()`；基准脚本新增 `_warmup_models()` 在任务循环前调用。

## 六、短输出（SHORT）阈值修正

**改动**：下调 analyst 与 report_writer 的 SHORT 检测阈值。

**原因**：SHORT 阈值用于**检测截断/乱码等无效输出**，而非强制质量下限。v0.1.3 的阈值定得过高，把正常的简短有效回答误判为失败并触发重试。

**会导致什么**：误触发的重试浪费一次完整 LLM 调用。典型案例：`report_writer_deep` 阈值 5345 远超模型实际输出（3500-4200 字），导致 57.5s 重试、总耗时 88s（应为 ~18s）。

**怎么改的**：阈值下调到只检测明显截断的水平：

| 节点 | v0.1.3 | v0.1.4 |
|------|--------|--------|
| `analyst` | 276 | 150 |
| `report_writer_comparison` | 3514 | 800 |
| `report_writer_deep` | 5345 | 1500 |

修正后 `report_writer` 耗时 88s → 18s，SHORT 误触发归零。

## 七、基准测试工具

**改动**：新增 `benchmark/benchmark_rag.py`（546 行）。

**原因**：v0.1.3 缺少对 RAG 各阶段质量的横向评测手段。

**会导致什么**：可对 12 个任务批量评测切片质量（P50/P90 字数、碎片率）、检索质量（Qdrant 分、score_std、阈值通过率、来源多样性、多源率）、证据缺口率，可选 RAGAS LLM-judge（faithfulness / context_precision）。

**用法**：

```bash
python benchmark/benchmark_rag.py                 # 全部 12 任务
python benchmark/benchmark_rag.py 05 07 09        # 指定任务
python benchmark/benchmark_rag.py --skip-ragas    # 跳过 RAGAS，只看确定性指标
python benchmark/benchmark_rag.py --save          # 结果写入 benchmark/results/
```

## 九、已知遗留问题

1. **score_std 绝对值仍偏低**（0.013-0.015）：切片修复使区分度方向性提升，但 ~750 中文字仍略超 512-token 窗口，bge-m3 对主题集中语料打分本身偏挤。可选后续：`chunk_size` 降至 512，或 `max_seq_length` 提至 768。
2. **来源多样性提升存在混淆**：planner 非确定（两次跑子问题数不同）+ Tavily key 可用性差异，使多样性提升无法纯粹归因于切片修复，方向正确但不宜夸大。
3. **Tavily 配额**：key 在单任务中途耗尽仍会拖慢 retriever，本质需补充可用 key，非代码节流可根治。
