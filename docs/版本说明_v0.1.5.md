# 版本说明 v0.1.5

> 发布日期：2026-06-28 · 上一版本：[v0.1.4](版本说明_v0.1.4.md) · 对比基线：commit `9866e22`

本版本单一主线：**系统性提升 RAG 检索质量**。先建立确定性可复现的检索基准，再以数据驱动逐项优化，最终把生产检索的 **Recall@6 从 0.800 提升到 0.892（+9.2pp）**。过程中坚持"按自己的冻结基准 A/B、记录负结果、不为差数据聚类就归因"的方法论，多个看似合理的方案（sparse 混合检索、语料去重）经实测被否决并归档。

## 二、检索质量基准框架（地基）

**改动**：新增 `benchmark/build_golden_set.py`、`benchmark/eval_retrieval.py`、`benchmark/GoldenDataset/`。

**原因**：v0.1.4 的 `benchmark_rag.py` 测的是切片/检索分布等"过程指标"，缺少**业内标准的端到端检索质量指标**（Recall@k / MRR / NDCG@k），无法回答"某次改动到底有没有提升召回"。

**怎么做的**：
- `build_golden_set.py`：从 12 个真实任务语料中 LLM 合成 120 道黄金问题（每任务 10 题），每题标注 gold chunk，冻结为 `golden_set.json`。
- `eval_retrieval.py`：在内存向量库中**复刻生产检索路径**（向量 top-40 → CrossEncoder rerank → top-6），对每题计算 gold 命中排名，输出 Recall@k / MRR / NDCG@k，并分离**两类损失**：向量漏检（gold 没进候选池）vs rerank 截断（进了池子被挤出 top-6）。全程零网络、零 LLM、**确定性可复现**。

**纪律**：冻结的 `golden_set.json` 保证每次 eval 在同一批 120 题上确定性可比；**对比优化只看 120 题汇总**（逐任务 10 题方差 ±10~20pp），且**绝不在对比前重建黄金集**（重建会重新生成问题、引入 ±3pp 噪声）。

> 完整诊断与方案演进（含否决方案）见 [`benchmark/GoldenDataset/improvement_plan.md`](../benchmark/GoldenDataset/improvement_plan.md)。

## 三、源头 markdown 清洗

**改动**：新增 `app/services/markdown_cleaner.py`，接入 `TextChunker._clean_chunks`。

**原因**：Tavily `raw_content` 是 markdown 化整页正文，含大量导航/页脚/引用锚点等噪声，污染 chunk 的 embedding。

**怎么做的**：
- `md_to_text()`：markdown → HTML → `BeautifulSoup.get_text`。`<a>` 保留锚文本丢 href、`<img>` 丢弃、代码块/表格正文完整保留（正则做不到——会误删代码块里的 `[x](y)`）。
- `is_boilerplate()`：**只按"剥离标记后塌缩率"判样板**（导航/图墙留存极低、表格/正文不塌缩），保守阈值全语料丢 ~2%、零误杀数值表格。
- 单一插入点，生产切片与黄金集构建共用同一清洗逻辑（同处净化生产证据链与采样池）。

**效果**：对问题措辞不敏感的两项稳健提升——**R@1 +11.7pp（0.300→0.417）**、向量上限 **Recall@20 +2.5pp（0.800→0.825）**；机理是把 URL/导航污染从 embedding 里清掉，gold 排得更靠前。

> 调研后明确**不做**（避免镀金）：图片 OCR（审计 3173 张图，真 OCR 候选仅 118 张、占 3.7%，且 web-markdown ≠ 文档智能 RAG）；清洗器换 AST/`unstructured`/`trafilatura`（已与极简正则 99.36% 一致，非瓶颈）。

## 四、扩大向量召回候选池（retrieve_k 20→40）

**改动**：`reranker_retrieve_k` 默认 `20 → 40`。

**原因**：向量召回阶梯 `@10=0.733 → @20=0.825` 未饱和，提高 k 仍能把更多 gold 召进候选池供 reranker 提取。

**效果**：**Recall@6 0.808→0.858（+5.0pp）**，Recall@40 达 0.892，硬漏检 21→13。**代价**：候选池翻倍 → rerank 算 40 对（≈2× rerank 延迟）；研究系统（非实时）可接受。

## 五、跨语言检索（本版核心）

**根因（多维分析实测）**：对 120 题做 [问题 × gold × 任务] 多维交叉后定位——失败题 **94% 是"中文问题 → 英文 gold"**：同语言（中问中gold）命中 98%，但"中问 + 纯英文 gold + 无共享 ASCII 锚点"塌到 67%。即 **bge-m3 dense 在 zh→en 无词面锚点时跨语言对齐失效**，是 Recall 的主要瓶颈，而非精确词、重复块或 rerank。

**改动**：新增 `app/services/translation_service.py`（本地 MT 译写），`rag_service.retrieve_evidence` 支持中英双路检索，新增 `xling_enabled` / `translation_model` 配置。

**怎么做的**：
- `xling_enabled=true` 且 query 含中文时，用本地 MT（`Helsinki-NLP/opus-mt-zh-en`）把 query 译成英文作为**第二路**；中英双路各 dense 检索 top-k，**按 cid 合并取最高分**后送 reranker。英译 query 把英文 gold 召进候选池，多语言 reranker 再排进 top-6。
- 翻译走**独立本地小模型**（opus-mt，77M/~300MB，**20ms/题**，懒加载 + 串行推理），**不占用主 LLM 并发闸门**；翻译失败优雅降级为单路检索。

**效果**：**Recall@6 0.858→0.892（+3.4pp）**，硬漏检 13→7。救回的恰好是"具体中文问题→英文 gold"的真跨语言案例；**残留 7 条漏检几乎全是宽泛/单 gold 假象题**（如"这篇文章涉及哪个编程语言"），属测量假象而非检索缺陷。

**选型说明**：曾对比 LLM（mimo）翻译——opus-mt 与之 Recall@6 **完全持平（均 0.892）**，但延迟 20ms vs 秒级、且零主 LLM 负载。"换更强多语言 embedding 免翻译"路线评估后搁置（bge-m3 已是开源多语言 SOTA，且系统为动态语料，翻 query 远比翻整批文档省）。

## 六、走过的弯路与负结果（实测归档，避免后人重蹈）

本版一半的工作是**排除错误方向**。完整记录被证伪的方案与误判的根因，是本版的方法论核心。

### 6.1 误判的根因（方法论纠偏）

定位瓶颈本身走了弯路，每一步"想当然的根因"都被数据推翻：

| 一度认定的根因 | 推翻它的实测 |
|----------------|--------------|
| 「精确词/实体（H200、PIPL）是 dense 弱点」→ 拟上 sparse | sparse 实测 **−0.8pp**；且 `q_ascii` 在漏检里**偏低**（0.23 vs 命中组 0.70），含 ASCII 实体的问题大多命中 |
| 「瓶颈在 rerank 截断」→ 拟调 reranker | rerank 截断仅 3.3%（4 条），**向量漏检 10.8%（13 条）才是大头**；逐条看 reranker 其实工作正常 |
| 「rerank 截断是重复块占槽」→ 拟 dedup | `twin=0`（无一条由孪生顶替 gold）、dup 率**跨任务不相关**——纯属"差数据恰好聚在高 dup 的 task04"的错觉 |
| **正解：跨语言**（多维交叉后定位） | 失败题 **94% 是「中文问题→英文 gold」**；跨语言译写 +3.4pp 直接验证 |

> **教训**：差数据聚在某一类（某任务、某题型），不等于那一类就是病因。必须做 [问题 × gold × 任务] 多维交叉、看跨维度是否一致，才能避免"聚类即归因"。

### 6.2 否决的技术方案（均有实测）

| 方案 | 实测结果 | 否决原因 |
|------|----------|----------|
| **Sparse 混合检索**（fastembed BM25 + Qdrant 原生 RRF 融合） | Recall@6 0.858**→0.850（−0.8pp）** | ① BM25 英文分词器无中文切分，"H200相比"糊成一个 token、query 仅切出 2 词、sparse 命中 0 文档；② 补 ASCII 隔离后能把 H200 检索 rank 38→17，但**净值仍 −0.8pp**——RRF 反挤掉 dense 已召回的 gold（Recall@40 0.892→0.883）；③ 目标 4 条"B 类"实为 rerank 截断、gold 早在候选池——sparse 解决的是"没进池"，是错的工具。跨语言主因下 BM25 更是**设计上无效**（中文 query token 永不匹配英文 doc token）。 |
| **语料去重（dedup）** | 非根因，未采纳 | 见 6.1：`twin=0`、dup 率跨任务不相关。最多修 2/120 rerank 截断，降为可选 hygiene。 |
| **换更强多语言 embedding** | 评估后搁置 | bge-m3 已是开源多语言 SOTA；确定性更优者需 7B LLM 底座或商用 API（成本高）；且系统为**动态语料**（每次研究现抓现建索引），翻 1 个 query ≪ 翻整批新抓文档，**翻 query 才是 LLM 最省路径**。 |
| **索引期翻译文档**（document translation） | 否决 | 动态语料下=每轮翻上千块，比翻 query 重得多。 |

相关 hybrid/sparse 探索代码已在本版完成后**清理回退**（`vector_store.py` 逐字节还原），仅保留结论于 `improvement_plan.md`。

## 八、已知遗留问题 / 后续

1. **残留 7 条硬漏检疑为单 gold 假象**：宽泛枚举题（"关键技术概念有哪些"）语料中多块可答、只标一个 gold，被判 miss。下一步可做**轻量 P2 多 gold 标注**量化假象规模、校准真实天花板（~0.92+），确认后即可收手。
2. **跨语言默认关闭**：`xling_enabled` 默认 `false` 以兼容无 opus-mt 模型的部署环境，生产需在 `.env` 显式开启并确保模型可下载。
3. **eval 翻译为实时调用**：`eval_retrieval.py` 在 `XLING_ENABLED=1` 下走生产 TranslationService 实时翻译（beam search 确定性，结果可复现），无独立冻结译文文件。
