# Deep Research Agent — 开发者上手指南

> 适用版本：v0.1.3+  
> 面向：首次加入项目的开发者

---

## 1. 项目是什么

用户输入一个研究问题，系统自动完成：

```
问题分解 → 联网搜索 → 网页抓取 → 信源评估 → RAG 证据构建
→ 子问题并发分析 → 事实核查 → 生成 Markdown 研究报告
```

技术栈：**LangGraph**（工作流编排）+ **FastAPI**（HTTP 服务）+ **Qdrant**（向量存储）+ **LiteLLM**（统一 LLM 接口）+ **fastembed**（本地 Embedding）。

## 2. 环境准备

### 2.1 Python 环境

项目使用固定 Conda 环境，**所有命令必须用这个解释器**：

```
D:\conda\envs\deepresearch\python.exe
```

激活方式：
```bash
conda activate deepresearch
```

### 2.2 配置文件

复制 `.env.example` 为 `.env`，必填项：

```env
LLM_API_KEY=你的密钥
LLM_BASE_URL=https://...   # 若用 OpenAI 兼容 API（如小米 MiMo）

SEARCH_PROVIDER=duckduckgo  # 本地开发免费，无需 key
# 或
USE_TAVILY=true
TAVILY_API_KEY=tvly-xxx     # Tavily 首个 key；多 key 轮换见 tavily.txt
```

**Tavily 多 key 轮换**：在项目根目录创建 `tavily.txt`，每行一个 key，系统自动顺序轮换（文件已加入 `.gitignore`，不会提交）。

D:\conda\envs\deepresearch\python.exe -m uvicorn app.main:app --reload --port 8000

## 5. 核心架构：LangGraph 流水线

### 5.1 节点顺序

```
planner → retriever → content_extractor → source_evaluator
        → evidence_builder → analyst → fact_checker → report_writer → END
```

**所有边都是固定顺序边**，没有条件分支。多轮研究（补充搜索）不走 LangGraph 的条件边，而是由 `routes_research.py` 中的 `_run_research()` 外部 while 循环控制。

> 注意：`nodes.py` 末尾的 `should_continue_research()` 函数是**死代码**，未被 `builder.py` 引用，不要误用。

### 5.2 各节点职责速览

| 节点 | 输入 | 输出 | LLM? |
|------|------|------|------|
| **planner** | `query` + `user_hints` | `sub_questions`、`research_strategy` | ✅ |
| **retriever** | `search_queries` | `search_results`、`search_summaries` | ❌ |
| **content_extractor** | `search_results` | `crawled_documents` | ❌ |
| **source_evaluator** | `crawled_documents` | `evaluated_sources` | ❌ |
| **evidence_builder** | `evaluated_sources` + docs | `evidence_chunks`（Qdrant） | ❌ |
| **analyst** | `sub_questions` + evidence | `sub_answers`、`citation_registry` | ✅（并发） |
| **fact_checker** | `sub_answers` + docs | `fact_check_result`、`follow_up_queries` | ✅（并发） |
| **report_writer** | `sub_answers` + registry | `final_report`（Markdown） | ✅ |

### 5.3 ResearchState

流水线中所有节点共享一个 `ResearchState`（`app/graph/state.py`），TypedDict 类型。每个节点返回一个 dict，LangGraph 自动合并到 state 中。

## 6. 关键设计决策（非代码显而易见）

### 6.1 Analyst 上下文分层与缓存命中

`analyst_node` 为所有子问题构建**相同的 system message**（包含完整证据文本），每个子问题只在 user message 中追加自己的特定内容。这使得 LLM 后端能对共享的 system message 前缀建立 KV cache，后续调用直接命中，节省 prompt 处理时间。

```python
# 所有子问题共享相同 system message → 缓存命中
messages = [
    {"role": "system", "content": shared_system_content},   # 相同
    {"role": "user",   "content": per_question_content},    # 各异
]
```

### 6.2 Citation Registry（引用注册表）

`analyst_node` 在开始分析前为每篇通过评估的来源分配唯一稳定 ID（C01、C02…），写入 `state["citation_registry"]`。这个 ID 在所有子问题和报告中保持一致，fact_checker 和 report_writer 都基于此做来源关联。

### 6.3 Short Output 重试机制

所有 LLM 节点调用通过 `_llm_with_short_retry()`，首次输出低于字符阈值时，会将原始输出 + 诊断信息注入 multi-turn 消息重试一次。阈值定义在 `nodes.py` 顶部的 `_SHORT_THRESHOLDS`。

### 6.4 Qdrant 始终是内存模式

`VectorStoreService` 使用 `AsyncQdrantClient(location=":memory:")` 硬编码，即使 `.env` 配置了 `QDRANT_URL` 也不切换到远程。向量数据在进程重启后全部丢失。要切换持久化，需修改 `vector_store.py` 的构造函数。

### 6.5 Tavily key 轮换逻辑

- quota 耗尽（HTTP 432）→ `TavilyQuotaError` → 永久轮换到下一个 key
- 限速（HTTP 429 / "exceeded rate limit"）→ `TavilyRateLimitError` → 仅短暂等待，不轮换
- 检测顺序：**先判限速，再判配额**，避免 "exceeded" 关键词被误归入配额耗尽

### 6.6 任务持久化

`TaskService` 用内存字典 + JSON 文件（`output/tasks/<task_id>.json`）双写。`list_tasks()` 只返回当前进程内存中的任务；直接调用 `get_task(task_id)` 时才从磁盘恢复。

## 7. 提示词修改

提示词以 Markdown 文件存放在 `app/prompts/`，可以直接编辑，**无需修改 Python 代码**。文件不存在时节点自动回退到 `nodes.py` 中的内置默认提示词。

修改提示词后重启服务立即生效，不需要重新部署。

## 8. 如何新增或修改节点

**修改现有节点**：直接编辑 `nodes.py` 中对应的 `xxx_node` 函数，返回一个 dict 更新 `ResearchState` 中的字段。

**新增节点**：
1. 在 `nodes.py` 中实现 `async def new_node(state: ResearchState) -> dict`
2. 在 `builder.py` 中 `add_node("new_node", new_node)`
3. 用 `add_edge` 插入到流水线的正确位置
4. 在 `state.py` 中为节点输出的新字段添加类型注解
5. 在 `routes_research.py` 的 `NODE_LABELS` 和 `NODE_PROGRESS` 中添加进度信息

## 14. 常见 Q&A

**Q：为什么我的 Tavily key 一直在轮换？**  
A：检查 `search_service.py` 的错误分类逻辑——"exceeded rate limit" 里的 "exceed" 会被正确识别为限速（不触发轮换），但若看到 quota 关键词（"usage limit"、"credits"）则轮换。真正耗尽的 key 应从 `tavily.txt` 删除。

**Q：分析结果为什么有时没有引用来源？**  
A：可能是 evidence_builder 阶段 Qdrant 为空（RAG 未命中），analyst 回退到文档全文模式，citation 质量降低。可查 `analyst` 节点日志中 `共享 RAG 池` 的 chunk 数量。

**Q：如何调整子问题数量？**  
A：修改 `app/prompts/planner.md` 中的指令，或在 `ResearchTaskCreate` 中设置 `report_type=summary`（轻量模式，子问题更少）。

**Q：如何让报告用英文输出？**  
A：请求时设置 `"language": "en"`，planner 会相应调整搜索语言。
