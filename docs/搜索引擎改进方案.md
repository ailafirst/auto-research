# 搜索引擎改进方案

> 记录 `retriever_node` 并发结构演进、Tavily 搜索引擎接入，以及零结果概率降低与降级方案的完整改进历程。

---

## 一、Retriever 并发结构演进（V1 → V4）

详细的版本对比与 benchmark 数据见 `benchmark/retriever_improvement_report.md`。本节记录最终采用的 V4 方案及其设计决策。

### 核心问题

原始 V1（全部并发，无归属）虽然最快，但搜索结果是无结构的平列列表：下游 `analyst_node` 无法知道哪条结果对应哪个子问题，报告章节与子问题无法对应。

V2（串行子问题）引入归属但损失 3.4× 速度；V3（并发子问题 + 共享 Semaphore）恢复了速度，但代码结构有额外的分组层次。

### V4：展平时打标（当前实现）

```
V1 思路:  flat_queries → gather(*all) — 快，但无归属
V3 思路:  sub_questions → gather(*per_sq(multi_search)) — 有归属，嵌套
V4 思路:  展平时同时携带 sq_id → gather(*all_tagged) — 快 + 有归属
```

**实现（`nodes.py — retriever_node`）：**

```python
tagged: list[tuple[str, str]] = [
    (q, sq.get("id", ""))
    for sq in sub_questions
    for q in sq.get("search_queries", [])
]

async def _search_tagged(query: str, sq_id: str) -> list[dict[str, Any]]:
    results = await search_service.search(query, max_results=settings.max_search_results)
    return [{**r.model_dump(), "sub_question_id": sq_id} for r in results]

batches = await asyncio.gather(
    *[_search_tagged(q, sq_id) for q, sq_id in tagged],
    return_exceptions=True,
)
```

**关键洞察**：归属信息在**展平时**注入，无需在查询时维护分组结构。全部查询同时并发（等同 V1 速度），每条结果天然携带 `sub_question_id`。

| | V1（原始） | V3（并发SQ+Semaphore） | V4（当前） |
|--|------------|----------------------|------------|
| 平均耗时/题 | 8.2s | 14.5s | ~8.2s |
| sub_question_id | ✗ | ✓ | ✓ |
| 代码复杂度 | 简单 | 嵌套 Semaphore | 简单 |
| 并发峰值 | 无限制 | ≤5（Semaphore） | 无限制（由引擎层控制） |

V4 的限速控制下移到 `SearchService` 层（`_tavily_semaphore` + DDG 自身限速），不再在 retriever 层维护 Semaphore。

## 二、Tavily 搜索引擎接入

### 2.1 USE_TAVILY 开关

通过 `.env` 中 `USE_TAVILY=true/false` 切换主引擎，默认启用 Tavily：

```python
# app/core/config.py
use_tavily: bool = True    # true=Tavily，false=DuckDuckGo

# app/services/search_service.py
self._current_provider = "tavily" if settings.use_tavily else "duckduckgo"
```

**设计原则**：不是系统级切换，是"主引擎"选择。零结果时的降级行为（见第三节）独立于此开关。

### 2.2 raw_content — 跳过 HTTP 爬取

Tavily API 在搜索时同步返回 `raw_content`（完整页面正文，通常 10000–40000 chars），可完全跳过 `content_extractor_node` 的 HTTP 爬取步骤。

**实现（`nodes.py — content_extractor_node`）：**

```python
for r in search_results:
    raw = r.get("raw_content")
    if raw:
        # Tavily 已提供正文，直接用，fetch_time=0
        documents.append({
            "url": r["url"], "title": r.get("title", ""),
            "content": raw, "text_length": len(raw),
            "fetch_time": 0.0, "error": None,
            "tavily_score": r.get("tavily_score"),
        })
    elif r.get("url"):
        urls_to_crawl.append(r["url"])   # 无 raw_content 才走爬虫

if urls_to_crawl:
    crawler = CrawlerService()
    crawled = await crawler.batch_fetch(urls_to_crawl)
    documents.extend([doc.model_dump() for doc in crawled])
```

**实测效果**（3题benchmark）：raw_content 覆盖率 100%，`content_extractor` 耗时从 ~15s 降至 ~0s。

**RAG 上下文不爆炸的原因**：raw_content（40k chars）被 `TextChunker`（chunk_size=800）切片后存入 Qdrant，`analyst_node` 仅检索 top_k=6 片段（≈4800 chars），LLM 上下文体积有界。

### 2.3 tavily_score — 替换启发式评分

原 `source_evaluator_node` 用 `content_length / 3000` 作为相关性代理，并叠加两个固定常量：

```python
# 旧实现（死常量）
credibility = 0.7   # 所有文档相同
freshness   = 0.8   # 所有文档相同
final_score = relevance * 0.5 + credibility * 0.3 + freshness * 0.2
# 展开后 = (content_length/3000) * 0.5 + 0.37
# → 任何 content_length > 100 chars 的文档几乎都通过
```

新实现直接用 Tavily 返回的 `score`（0–1，基于 Tavily 内部语义相关度模型）：

```python
tavily_score = doc.get("tavily_score")
if tavily_score is not None:
    final_score = round(tavily_score, 3)
    accepted = final_score >= 0.5 and content_length > 50
    reason = f"Tavily score={final_score}"
else:
    # DDG 降级或无 score 时，回退到长度启发
    final_score = round(min(1.0, content_length / 3000), 3)
    accepted = content_length > 300
    reason = "内容充足" if accepted else f"内容过短 ({content_length} chars)"
```

`credibility` 和 `freshness` 两个固定常量**完全移除**。

### 2.4 query_answer → search_summaries — 注入 analyst 上下文

Tavily 为每次搜索调用生成一条 AI 摘要（`answer` 字段，对应该次查询的综合回答）。

**收集路径（`retriever_node`）：**

```python
summaries: list[dict[str, Any]] = []
seen_answers: set[str] = set()
for r in deduped:
    answer = r.get("query_answer")
    sq_id = r.get("sub_question_id", "")
    if answer and answer not in seen_answers:
        seen_answers.add(answer)
        summaries.append({"sq_id": sq_id, "answer": answer})
# → state["search_summaries"]
```

`query_answer` 是 per-search-call 级别（同一次 Tavily 调用的所有结果共享同一 answer），去重后存入 `search_summaries`。

**注入路径（`analyst_node — _analyze_single_question`）：**

```python
sq_answers = [s["answer"] for s in search_summaries if s.get("sq_id") == qid]
if sq_answers:
    bullets = "\n".join(f"- {a}" for a in sq_answers[:3])
    context_material.append(f"[Tavily 搜索摘要]\n{bullets}")
```

每个子问题分析时，LLM 同时看到：① RAG 检索片段 ② Tavily AI 摘要。

**`search_summaries` 状态字段**（`app/graph/state.py`）：

```python
search_summaries: list[dict[str, Any]]   # [{sq_id, answer}]
```

需在所有状态初始化处同步添加 `"search_summaries": []`（`routes_research.py`、`main.py`、`benchmark/` 相关文件）。

## 三、零结果降低方案

### 3.1 根因分析

Tavily dev key（`tvly-dev-*`）在高并发下会触发限速，返回 "Your request has been blocked due to excessive requests"。原实现存在两个缺陷：

**缺陷一：retry 完全失效**

`TavilyProvider.search()` 把所有异常都吞掉返回 `[]`，`@retry` 装饰器在 `SearchService.search()` 上永远看不到异常，不会触发任何重试。

**缺陷二：Semaphore 无间隔，仍有突发**

`asyncio.Semaphore(3)` 限制并发数，但 3 个请求同时完成后下一批 3 个立即开始——仍然是时间上的突发请求。

**缺陷三：零结果无兜底**

Tavily 返回 `[]` 时，该查询直接贡献零结果，retriever 静默丢弃，无任何补救。

### 3.2 三层防御实现

```
TavilyProvider.search()
  ├─ asyncio.Semaphore(3)：最多 3 个并发请求
  ├─ asyncio.sleep(0.3) finally：每次请求后强制间隔，防连续突发
  └─ 限速关键词检测 → 上抛 TavilyRateLimitError（不再吞掉）
         关键词: "excessive", "rate", "blocked", "429", "too many"

SearchService._primary_search()  ← @retry 在此层生效
  └─ TavilyRateLimitError → 指数退避重试（最多 3 次）
       第1次失败 → 等 2s → 第2次
       第2次失败 → 等 4s → 第3次（最长 30s）
       第3次失败 → reraise=True → 上抛给 search()

SearchService.search()  ← 最终降级在此层
  ├─ 捕获 TavilyRateLimitError（3次耗尽）→ results = []
  └─ results == [] → 自动调用 DuckDuckGoProvider.search()
       DuckDuckGo 结果作为最终兜底，透明对上层
```

**`TavilyRateLimitError`（新增自定义异常类）：**

```python
class TavilyRateLimitError(Exception):
    """Tavily 限速错误 — 可被 tenacity 捕获并按指数退避重试。"""
```

**`TavilyProvider.search()` 关键修改：**

```python
except Exception as exc:
    msg = str(exc)
    if any(k in msg.lower() for k in ("excessive", "rate", "blocked", "429", "too many")):
        raise TavilyRateLimitError(msg) from exc   # 上抛，可重试
    logger.error("Tavily 搜索失败: %s", exc)
    return []                                       # 其他错误静默
finally:
    await asyncio.sleep(0.3)   # 每次请求后强制间隔
```

**`SearchService` 分离重试与降级：**

```python
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=2, min=2, max=30),
    retry=retry_if_exception_type(TavilyRateLimitError),
    reraise=True,
)
async def _primary_search(self, query: str, max_results: int) -> list[SearchResult]:
    return await self.provider.search(query, max_results=max_results)

async def search(self, query: str, max_results: int | None = None) -> list[SearchResult]:
    n_results = max_results or settings.max_search_results
    try:
        results = await self._primary_search(query, n_results)
    except TavilyRateLimitError:
        logger.warning("Tavily 限速重试耗尽，降级 DDG: query='%s'", query[:50])
        results = []

    if not results and self._current_provider == "tavily":
        logger.warning("Tavily 零结果，DDG 降级: query='%s'", query[:50])
        results = await self.providers["duckduckgo"].search(query, max_results=n_results)

    return results
```

### 3.3 设计决策

**降级是 per-query 的，不是全局切换**

DDG 降级只在单次查询零结果时触发，不影响其他并发查询继续用 Tavily。与 `USE_TAVILY` 开关（全局选择主引擎）完全独立：

```
USE_TAVILY=true:   所有查询先用 Tavily → 单条零结果时降 DDG
USE_TAVILY=false:  所有查询直接用 DDG，无降级逻辑
```

**`asyncio.sleep(0.3)` 在 finally 而非外部**

放在 Semaphore 内的 `finally` 中，确保即使请求成功也执行间隔，且间隔时间叠加在 Semaphore 等待上——任意时刻全局 Tavily 请求速率上限约为 `3 / 0.3 = 10 QPS`，适合 dev key 限制。

**retry 只针对 `TavilyRateLimitError`，不针对所有异常**

网络超时、JSON 解析失败等不应重试（重试无效）；限速错误（瞬时状态）才应等待后重试。`retry_if_exception_type(TavilyRateLimitError)` 精确限定范围。

### 3.4 验证

三条路径独立测试通过：

| 场景 | 行为 | 结果 |
|------|------|------|
| 正常 Tavily 搜索 | 直接返回 Tavily 结果 | 2 条（provider=tavily） |
| Tavily 持续限速（3次重试耗尽） | 捕获 `TavilyRateLimitError` → DDG | 2 条（DDG 兜底） |
| Tavily 返回空结果（非限速） | `results==[]` → DDG | 2 条（DDG 兜底） |

## 四、多轮研究 Retriever 修复（V0.1.2）

### 4.1 问题：第 2 轮搜索与第 1 轮完全重复

修复前，`routes_research.py` 构建第 2 轮 state 时将 `sub_questions` 重置为空，导致：

1. `planner_node` 对原始 `query` 重新规划 → 生成与第 1 轮几乎相同的子问题和搜索词
2. `follow_up_queries`（fact_checker 识别的补充方向）传入 `search_queries` 后被 planner 覆盖 → 从未被实际搜索
3. `retriever_node` 重复搜索第 1 轮相同关键词 → 结果高度重叠，无增量信息

### 4.2 修复方案

**`routes_research.py`（第 2 轮 state 构建）**：

```python
new_state = _build_state(current_round, {
    "sub_questions":     final_state.get("sub_questions", []),    # 复用第 1 轮
    "research_plan":     final_state.get("research_plan", {}),
    "research_strategy": final_state.get("research_strategy", {}),
    "follow_up_queries": follow_up,                               # 保留供 retriever 使用
    "search_queries":    follow_up[:5],
    "search_results":    [],
    ...
})
```

**`retriever_node`（`nodes.py`）**：检测到第 2 轮 + follow_up 非空时，只搜索 follow_up 查询：

```python
follow_ups = state.get("follow_up_queries", [])
if follow_ups and state.get("current_round", 1) > 1:
    tagged = [(q, "") for q in follow_ups[:5]]   # sq_id="" → analyst 走 accepted_docs 全局回退
else:
    tagged = [(q, sq_id) for sq in sub_questions for q in sq["search_queries"]]
```

follow_up 结果的 `sub_question_id` 为空，`analyst_node` 的 `_analyze_single_question` 在 RAG 无命中时自动回退到 `accepted_docs`（所有通过评估的文档），因此第 2 轮的新文档对所有子问题分析均可见。

### 4.3 效果对比

| | 修复前 | 修复后 |
|--|--------|--------|
| 第 2 轮 planner | 重新 LLM 规划（~5–10s，结果近似第 1 轮） | 跳过，直接复用第 1 轮计划 |
| 第 2 轮 retriever | 重复搜索第 1 轮相同关键词 | 搜索 fact_checker 识别的补充方向 |
| follow_up_queries | 传入但被 planner 覆盖，实际未使用 | 由 retriever 执行，获取增量证据 |

## 五、LLM 调用超时防护（V0.1.2）

### 5.1 问题：任务永久卡死

`analyst_node` 完成（进度 75%）后，`fact_checker_node` 调用 `LLMService.chat()` → 内部 `litellm.acompletion()` 无超时限制。若 LLM 服务响应慢或无响应，`acompletion` 永远不返回，整个 `asyncio.gather` 等待链无法推进，任务永久停在 75%（不会 failed，只会卡死）。

### 5.2 修复

```python
# app/services/llm_service.py
response = await litellm.acompletion(**kwargs, timeout=120)
```

加 `timeout=120`（秒）后，120s 无响应 → LiteLLM 抛出超时异常 → tenacity 最多重试 3 次 → 耗尽后异常传出节点 → LangGraph `astream` 中断 → `_run_research` 外层 `except` 捕获 → `_fail_task()` 将任务标记为 `failed`。任务不再卡死，而是明确失败并记录错误信息。

## 六、遗留问题与后续方向

### Tavily Dev Key 限制

`tvly-dev-*` 前缀的 key 有严格的请求频率限制。`Semaphore(3)` + `sleep(0.3)` 可覆盖大多数场景，但极高并发（>30 查询/次）仍可能触发限速。生产环境切换 production key 后可放宽 Semaphore 值。

可考虑将 Semaphore 值通过 config 暴露：

```python
# app/core/config.py（待实现）
tavily_concurrency: int = 3   # dev key 建议 2-3，production key 可到 8-10
```

### DuckDuckGo 稳定性

DDG 作为降级目标本身也有不稳定问题（CAPTCHA、rate limit）。长期看若需要更可靠的免费兜底，可评估 Brave Search API（免费 2000次/月，质量优于 DDG）。

### Serper / Bing 第三引擎

`config.py` 中已有 `serper_api_key` 和 `bing_api_key` 配置项，但 `SearchService` 尚未实现对应 Provider。可按 `DuckDuckGoProvider` 模式扩展，约 30 行代码。
