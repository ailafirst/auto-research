# 搜索引擎改进方案

> 记录 `retriever_node`（`app/graph/nodes.py`）、`SearchService`（`app/services/search_service.py`）及相关节点的完整改进历程，按版本号组织。

---

## v0.1.0 — 初始版本

### 基本实现

- 搜索引擎：DuckDuckGo（同步接口，通过 `asyncio.to_thread` 包装）
- Retriever：全部查询完全并发，但结果无 `sub_question_id` 归属
- 无零结果降级，无错误重试

### 已知缺陷

- 搜索结果为无结构平列列表，`analyst_node` 无法知道哪条结果对应哪个子问题
- DuckDuckGo 偶发限速，无兜底机制
- 无 raw_content，所有 URL 均需 HTTP 爬取（`content_extractor` 耗时 ~15s）

## v0.1.2 — Tavily 接入 & Retriever V4 & 零结果防护

### 1. Retriever V4：展平时打标

**核心问题**：V1（全部并发，无归属）虽然最快，但无法将搜索结果关联回对应子问题；V2（串行子问题）引入归属但损失 3.4× 速度；V3（并发子问题 + Semaphore）恢复速度但代码嵌套复杂。

**V4 方案**：在展平时同时携带 `sq_id`，全部查询完全并发，每条结果天然携带 `sub_question_id`：

```python
tagged: list[tuple[str, str]] = [
    (q, sq.get("id", ""))
    for sq in sub_questions
    for q in sq.get("search_queries", [])
]

async def _search_tagged(query: str, sq_id: str) -> list[dict]:
    results = await search_service.search(query, max_results=settings.max_search_results)
    return [{**r.model_dump(), "sub_question_id": sq_id} for r in results]

batches = await asyncio.gather(
    *[_search_tagged(q, sq_id) for q, sq_id in tagged],
    return_exceptions=True,
)
```

**关键洞察**：归属信息在**展平时**注入，无需在查询时维护分组结构。

| | V1（原始） | V3（并发SQ+Semaphore） | V4（当前） |
|--|------------|----------------------|------------|
| 平均耗时/题 | 8.2s | 14.5s | ~8.2s |
| sub_question_id | ✗ | ✓ | ✓ |
| 代码复杂度 | 简单 | 嵌套 Semaphore | 简单 |
| 限速控制 | 无 | retriever 层 Semaphore | 下移至 SearchService 层 |

### 2. Tavily 搜索引擎接入

#### 2.1 主引擎切换

通过 `.env` 中 `USE_TAVILY=true/false` 切换主引擎，默认启用 Tavily：

```python
self._current_provider = "tavily" if settings.use_tavily else "duckduckgo"
```

#### 2.2 raw_content — 跳过 HTTP 爬取

Tavily API 同步返回 `raw_content`（完整页面正文，通常 10000–40000 chars），可完全跳过 `content_extractor_node` 的 HTTP 爬取步骤：

```python
for r in search_results:
    raw = r.get("raw_content")
    if raw:
        documents.append({"url": r["url"], "content": raw, "fetch_time": 0.0, ...})
    elif r.get("url"):
        urls_to_crawl.append(r["url"])   # 无 raw_content 才走爬虫
```

**实测效果**：raw_content 覆盖率 100%，`content_extractor` 耗时从 ~15s 降至 ~0s。

#### 2.3 tavily_score — 替换启发式评分

`source_evaluator_node` 原来用 `content_length / 3000` 代理相关性，叠加两个固定常量（credibility=0.7，freshness=0.8）。现在直接使用 Tavily 返回的语义相关度分数：

```python
if tavily_score is not None:
    final_score = round(tavily_score, 3)
    accepted = final_score >= 0.5 and content_length > 50
else:
    # DDG 降级路径：回退到长度启发
    final_score = round(min(1.0, content_length / 3000), 3)
    accepted = content_length > 300
```

`credibility` 和 `freshness` 两个固定常量完全移除。

#### 2.4 search_summaries — Tavily AI 摘要注入 analyst

Tavily 为每次搜索调用生成一条 AI 摘要（`answer` 字段）。收集后按 `sq_id` 存入 `state["search_summaries"]`，注入 `analyst_node` 分析上下文：

```python
# retriever_node 收集
summaries: list[dict] = []
seen_answers: set[str] = set()
for r in deduped:
    answer = r.get("query_answer")
    if answer and answer not in seen_answers:
        seen_answers.add(answer)
        summaries.append({"sq_id": r.get("sub_question_id", ""), "answer": answer})

# _analyze_single_question 注入
sq_answers = [s["answer"] for s in search_summaries if s.get("sq_id") == qid]
if sq_answers:
    context_material.append(f"[Tavily 搜索摘要]\n" + "\n".join(f"- {a}" for a in sq_answers[:3]))
```

每个子问题分析时，LLM 同时看到：① RAG 检索片段 ② Tavily AI 摘要基线。

### 3. 零结果三层防护

#### 根因分析

Tavily dev key（`tvly-dev-*`）在高并发下触发限速，返回 "excessive requests" 错误。原实现存在三个缺陷：

1. **retry 完全失效**：`TavilyProvider.search()` 吞掉所有异常返回 `[]`，`@retry` 装饰器永远看不到异常
2. **Semaphore 无间隔仍有突发**：3 个请求同时完成后下一批立即开始
3. **零结果无兜底**：Tavily 返回 `[]` 时静默丢弃，无补救

#### 三层防御实现

```
TavilyProvider.search()
  ├─ asyncio.Semaphore(3)：最多 3 个并发请求
  ├─ asyncio.sleep(0.3) finally：每次请求后强制间隔（约 10 QPS 上限）
  └─ 限速关键词检测 → 上抛 TavilyRateLimitError（不再吞掉）
         关键词: "excessive", "rate", "blocked", "429", "too many"

SearchService._primary_search()  ← @retry 在此层生效
  └─ TavilyRateLimitError → 指数退避重试（最多 3 次，等待 2–30s）

SearchService.search()  ← 最终降级
  ├─ 捕获 TavilyRateLimitError（3次耗尽）→ results = []
  └─ results == [] → 自动调用 DuckDuckGoProvider.search() 兜底
```

#### 设计决策

- **降级是 per-query 的**，不影响其他并发查询继续用 Tavily
- **`sleep(0.3)` 在 finally 中**，确保即使请求成功也执行间隔，全局 Tavily 请求速率约 10 QPS
- **retry 只针对 `TavilyRateLimitError`**，网络超时等不重试

### 4. 多轮 Retriever 修复

#### 问题（修复前）

第 2 轮 `sub_questions` 被重置为空，Planner 重新规划同一 query，生成与第 1 轮几乎相同的搜索词；`follow_up_queries` 被 planner 覆盖从未使用。第 2 轮 retriever 等于重复第 1 轮。

#### 修复

第 2 轮 `retriever_node` 检测到 `follow_up_queries` 非空时，改用 fact_checker 识别的补充方向搜索，不重复第 1 轮关键词：

```python
follow_ups = state.get("follow_up_queries", [])
if follow_ups and state.get("current_round", 1) > 1:
    # 只搜索 follow_up，结果由 analyst accepted_docs 全局回退可见
    tagged = [(q, "") for q in follow_ups[:5]]
else:
    tagged = [(q, sq_id) for sq in sub_questions for q in sq["search_queries"]]
```

| | 修复前 | 修复后 |
|--|--------|--------|
| 第 2 轮搜索词 | 与第 1 轮相同（planner 重新规划） | follow_up_queries（增量方向） |
| follow_up_queries | 被 planner 覆盖，未使用 | 由 retriever 实际执行 |

### 5. LLM 调用超时防护

**问题**：`analyst_node` 完成（进度 75%）后，`fact_checker_node` 调用 `llm.chat()` → 内部 `litellm.acompletion()` 无超时限制，LLM 服务响应慢时任务永久停在 75%，不 failed 也不 complete。

**修复**（`app/services/llm_service.py`）：

```python
response = await litellm.acompletion(**kwargs, timeout=120)
```

超时 120s 后抛出异常 → 最多重试 3 次 → 耗尽后任务标记为 `failed`，不再卡死。

## Backlog — 待实现方向

### Tavily Dev Key 限制

`Semaphore(3)` + `sleep(0.3)` 可覆盖大多数场景，但极高并发（>30 查询/次）仍可能触发限速。可考虑将 Semaphore 值通过 config 暴露：

```python
# app/core/config.py（待实现）
tavily_concurrency: int = 3   # dev key 建议 2-3，production key 可到 8-10
```

### DuckDuckGo 稳定性

DDG 作为降级目标本身也有不稳定问题（CAPTCHA、rate limit）。长期可评估 Brave Search API（免费 2000次/月，质量优于 DDG）。

### Serper / Bing 第三引擎

`config.py` 中已有 `serper_api_key` 和 `bing_api_key` 配置项，但 `SearchService` 尚未实现对应 Provider。可按 `DuckDuckGoProvider` 模式扩展，约 30 行代码。
