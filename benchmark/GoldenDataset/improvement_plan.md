## 已完成：源头 markdown 清洗（2026-06-26）

### 背景与两次纠偏

最初两条方案是错的，已废弃：

- ~~**P0-1 黄金集采样过滤**~~（拒采含 `request was rejected` / `Star数量` / `引用[\d+]` 的 chunk）——**自欺欺人**：真实生产证据链里这些噪声照样存在，只在采样侧屏蔽等于粉饰基线，真实代码任务中出现就直接无效。
- ~~**P4 content_extractor boilerplate 强化**~~（基于「content_extractor 没滤掉反爬错误页」的假设）——**事实错误**：项目用的是 Tavily 自带 raw_content（markdown），`content_extractor_node` 只原样透传，不是 HTML 抽取器。

**正确做法**：噪声从源头止——在 Tavily 的 markdown raw_content 上做清洗，同一处既净化生产证据链、又净化黄金集采样池。

### 实现

- `app/services/markdown_cleaner.py`
  - `md_to_text()`：markdown → HTML → `BeautifulSoup.get_text`。`<a>` 天然留锚文本丢 href，`<img>` 丢弃，代码块/表格正文完整保留（正则做不到——会误删代码块里的 `[x](y)`）。
  - `is_boilerplate()`：**只按「剥离标记后塌缩率」判样板**——导航/图墙留存极低，表格/正文留存 ~100% 不塌缩。**绝不用「残留过短」独立条件**，那会误杀合法短正文与数值表格。保守阈值，全语料丢 ~2%、零误杀数值表格。
- 单一插入点 `TextChunker._clean_chunks`（`rag_service.py`）：生产切片与黄金集构建共用同一清洗逻辑。

### 实测效果（清洗前 → 清洗后冻结）

| 指标 | 清洗前 | 清洗后（冻结） |
|---|---|---|
| rerank Recall@6 | 0.800 | 0.808 |
| 向量上限 Recall@20 | 0.800 | 0.825 |
| **Recall@1** | **0.300** | **0.417** |
| MRR / NDCG@6 | 0.616 / 0.663 | 0.676 / 0.710 |
| 向量漏检 / rerank 截断 | 20.0% / 0.0% | 17.5% / 1.7% |

> **诚实结论**：最终 Recall@6 净增（+0.8pp）部分落在问题采样噪声带内，不宜过度解读。**真正稳健的是对问题措辞不敏感的两项——R@1（+11.7pp）与向量上限 Recall@20（+2.5pp）**，两次独立重建都一致提升。机理：把 URL/导航污染从 embedding 里清掉，gold 排得更靠前——是排序质量的实打实提升。

### 调研后明确「不做」（避免镀金）

- **图片 OCR**：审计 3173 张图 / 1559 唯一 URL——26% 是 SVG/logo/icon 装饰；1667 张「有意义 alt」的文本信号就是 alt 本身（保留 alt 即可，零成本）；真正的 OCR 候选只有 118 张 arXiv `Refer to caption` 图（占 3.7%，多为曲线图，文本产出低）。为这点边际收益搭 1500+ URL 的「抓取+OCR」子系统不划算，且 OCR 是「文档智能 RAG(PDF/扫描)」标准，不是 web-markdown RAG 的标准。**留待 eval 实证「有金块困在图里」再议。**
- **保留 alt 文本**（零成本小修补，覆盖 1667 张有意义图，与「留锚文本丢 href」一致）：技术正确，但字符级改动（均摊 3.7 字符/块）会被问题噪声淹没。**留到下次大改（P1）一并合入、一次重建一起测**，不单独为它重建。
- **清洗器换 AST 走树 / `unstructured` / `trafilatura`**：清洗已与「极简正则」99.36% 一致且差异是噪声，清洗不是瓶颈。继续给已达 99% 的环节加码属镀金，ROI 低于直接攻召回。

## 问题诊断（清洗后）

### 诊断结论

| 损失类型 | 比例 | 说明 |
|---|---|---|
| **向量漏检**（top-20 都没召回） | **17.5%** | 唯一瓶颈 |
| rerank 截断损失 | **1.7%（2条）** | 基本健康，非主要问题 |

rerank 把进入 top-20 的 gold 几乎全保留（仅 2 条被挤出 top-6），改进空间仍在**向量召回阶段**。

### 硬漏检分类（21 条 gold 未进 top-20）

清洗后 A 类噪声大幅减少（源头剥离），残留三类：

#### 残留噪声/弱问题（约 2 条）
保守 gate 放过了少量正文充足的导航块（如「网站导航的分类有哪些」）。gate 刻意保守以免误杀短正文/数值表格，故不追求 100% 清除——这是有意识的召回/精度权衡。

#### B 类：精确词/实体匹配（约 6-7 条）
数字+单位、产品代号、专有名词，dense embedding 对字面精确匹配天然弱：
- 「H200 相比 H100 的性能提升幅度是多少」
- 「采纳标准数据保护条款时依据哪些具体条款」
- 「固态电池技术演进第三阶段具体涉及哪些材料/技术」

**根因**：dense embedding 把语义相近向量聚在一起，对命名实体、数字+单位、产品代号的区分力弱——dense-only 检索的经典短板。**→ 原拟 P1 sparse 补强，但实测证伪（见「Hybrid Sparse — 不采纳」）：这些案例的 gold 多已在候选池内，真正卡在 rerank 阶段。**

#### C 类：语义跨度（约 12 条，本轮集中在任务 02/04）
query 与 gold 字面重叠低。注意**两种性质不能混淆**：
1. **真语义难**：embedding 抓不住 query↔chunk 深层关系。
2. **测量假象（可能占多数）**：黄金集构造要求「不照抄原句」，人为拉大字面距离；且**单 gold 标注系统性低估**——宽泛问题（「Rust 主要优势」「微服务好处」）语料中多个 chunk 都能答，只标 1 个为 gold，召回另一个同样正确的 chunk 被判 miss。**→ 需 P2 量化真实规模。**

> 逐类别计数为量级估计：受 10 题采样噪声影响，类别归属非精确（如 C 类上轮集中在 02/11，本轮移到 02/04）。

## 改进方案（聚焦检索质量）

### 优先级原则
> 先零成本探召回上限，再上混合检索主攻 B 类；先量化 C 类测量假象，再决定要不要 query 改写。

### P0-2 — 扩大 retrieve_k（✅ 已验证，+5.0pp）

召回阶梯 `@10=0.733 → @20=0.825` 未饱和，提高 k 仍有空间。`RERANKER_RETRIEVE_K=40` 跑 `eval_retrieval.py`（同一冻结黄金集、确定性可比，非噪声）：

| 指标 | k=20 | **k=40** | 变化 |
|---|---|---|---|
| 向量上限 Recall@k | 0.825 | **0.892** | +6.7pp |
| **rerank 后 Recall@6** | 0.808 | **0.858** | **+5.0pp** |
| MRR / NDCG@6 | 0.676 / 0.710 | 0.707 / 0.746 | +3.1 / +3.6pp |
| 向量漏检 / rerank 截断 | 17.5% / 1.7% | 10.8% / 3.3% | −6.7 / +1.6pp |
| 硬漏检条数 | 21 | **13** | −8 |

reranker 把大量 rank 21–40 的 gold 提进 top-6。**代价**：候选池翻倍 → rerank 算 40 对而非 20 对（≈2x rerank 延迟）；截断损失 1.7%→3.3%（正常，池子大了更多 gold 落在 reranker 提不动的位置）。阶梯 `@40=0.892` 仍未饱和，k=60 或可再榨 2~4pp 上限，但延迟与截断同步上升。

**建议**：固化 `reranker_retrieve_k=40`（一行配置）。为研究系统（非实时），+5pp 召回换 2x rerank 延迟划算。剩余损失里 **13 条向量漏检（gold 连 top-40 都没进，10.8%）才是大头**，另 4 条 rerank 截断仅 3.3%——根因见下「多维根因分析」（结论：主因是**跨语言漏检**，既非 rerank、也非重复块）。

### P1（❌ 已验证·负结果）：Hybrid Sparse 混合检索 — 不采纳

**假设**：dense 对精确词/实体（H200、PIPL、Qdrant）区分力弱，加 BM25 sparse 走 RRF 融合补强 B 类。

**实现**：fastembed `Qdrant/bm25`（纯分词器、零额外依赖）+ Qdrant 命名向量 + 原生 `FusionQuery(RRF)`；dense 路径完全不动（加性、`HYBRID_ENABLED` 开关守护）。代码已落地但默认关闭。

**实测（冻结黄金集，确定性可比，非采样噪声）**：

| 配置 | Recall@40 | rerank Recall@6 |
|---|---|---|
| dense-only K=40（基线） | 0.892 | **0.858** |
| + BM25 hybrid | 0.883 | 0.850 |
| + BM25 hybrid + ASCII 隔离 | 0.883 | 0.850 |

**两个变体都 −0.8pp。判定：不采纳。** 三层原因：

1. **BM25 无中文切分**：真实问题 `H200相比` 连写无空格，英文分词器把 H200 糊进中文 token，query 仅切出 2 个 token、sparse 命中 0 文档。加正则在 ASCII 串两侧补空格（H200/H100 隔离）后 sparse 能把 H200 检索 rank 38→17——机制对，但无济于事（见 3）。
2. **RRF 会挤掉 dense 已召回的 gold**：融合不是取并集，而是对 dense∪sparse 重排；BM25 命中的无关块（语料 161 块含 GPU）顶掉 dense 排 35-40 的 gold，Recall@40 0.892→0.883。sparse 噪声 ≥ 信号。
3. **根本错位（关键）**：那 4 条「B 类 exact-term」**根本不是检索漏检，是 rerank 截断**——gold 在纯 dense 下就已在 top-40 候选里（rank 38/11/2/21）。sparse 解决的是「gold 没进候选池」，而这些 gold 一直在池里。**瓶颈在 reranker 阶段，sparse 是错的工具。**

> route B（FlagEmbedding bge-m3 学习型 sparse）同理不采纳：能降 BM25 噪声，但救不了 rerank 瓶颈，还要加依赖——同样的瓶颈、更高成本。

## 多维根因分析（实测，2026-06-27）

> ⚠️ 方法论纠偏：先前只盯着 4 条 rerank 截断就归因「重复块/dedup」，犯了「差数据聚在某类就认定该类是病因」的错。下面对全部 120 题做 [问题 × gold × 任务] 多维交叉，结论与之相反。

### 损失分布（先看大头在哪）

| 失败阶段 | 占比 | 条数 |
|---|---|---|
| **向量漏检**（gold 没进 top-40） | **10.8%** | **13** |
| rerank 截断（进了 top-40 被切出 top-6） | 3.3% | 4 |

**大头是向量漏检（13 条），是 rerank 截断（4 条）的 3 倍。** dedup 只针对那 4 条里的 2 条 ≈ 1.7%，不是主战场。

### 逐维度命中率（找真正的判别维度）

| 维度 | 分组 | 命中率 |
|---|---|---|
| **gold 语言** | 中文 gold | **98%**（1 失败/43） |
| | 英文 gold | **79%**（16 失败/77） |
| **问题词重叠** | overlap≥0.05 | 92~100% |
| | overlap<0.05 | **67%**（14 失败/42） |
| 枚举/宽泛 | 枚举 vs 非枚举 | 80% vs 88%（弱） |

**失败题 94% 是「中文问题 → 英文 gold」**（对照全体 64%）。2×2 交叉：

| | overlap<0.05 | overlap≥0.05 |
|---|---|---|
| 英文 gold | **67% 命中**（14 失败） | 94% 命中（2 失败） |
| 中文 gold | （不存在） | 98% 命中（1 失败） |

中文问题对英文 token 天然零词重叠，「overlap≈0」与「跨语言」是同一拨题。关键：英文 gold **只要有一个共享 ASCII 锚点**（Qdrant/PIPL/RAG/H200，overlap≥0.05），命中率就从 67% 跳到 94%。

### 根本原因

**主因（约 10/17 失败）：跨语言语义鸿沟** —— 中文问题召回英文 gold 块、且无共享 ASCII 锚点时，bge-m3 dense 跨语言对齐失效。同语言（中问中gold）几乎不失败（98%），有锚点的跨语言也基本不失败（94%），**唯独「中问 + 纯英文 gold + 无锚点」塌到 67%**。

**次因（约 6/17）：单 gold 假象 / 宽泛低质问题** —— 「这篇文章涉及哪个编程语言」「网站指南的分类有哪些」「学习方式有哪两种」这类枚举题，语料里多块可答只标一个 gold。需 P2 才能从主因里分离。

### 被这次数据证伪的归因（均为「聚类即归因」之误）

| 先前说法 | 证伪数据 |
|---|---|
| dedup 是治本杠杆 | twin40/6 **全为 0**——无一条漏检由「孪生被召回、gold 没中」造成；dup 率与失败**跨任务不相关**（task12 dup 11.5% 仅 1 失败，task02 dup 1.4% 却 3 失败）。task04 高 dup+高失败是巧合。 |
| B 类 exact-term 是弱点 | q_ascii 在漏检里**偏低**（0.23 vs 命中 0.70）；英文 gold **有** ASCII 锚点时命中 94%。exact-term 问题大多成功。 |
| 瓶颈在 rerank | 仅 3.3%（4 条）且 reranker 工作正常；大头是向量漏检 10.8%。 |
| sparse/BM25 能补强 | 跨语言失败模式下 BM25 **必然无效**（中文 query token 永不匹配英文 doc token）。sparse 从设计上救不了主因——这也解释了 P1 为何负结果。 |

## 改进方案（v3，按跨语言根因重排）

### P-XLing — 跨语言 query 处理（主攻主因，✅ 已验证 +3.4pp）
**解决**：中问→英gold 无锚点的约 10 条漏检。
**怎么做**：检索时把中文 query 用 LLM 译成英文，原文 + 英译**双 query 各检索 top-k、按 cid 合并**后送 reranker；英译 query 把英文 gold 召进候选池，reranker（多语言，已验证能给跨语言对打高分如 PIPL 0.988）再排进 top-6。
- 生产侧：`retrieve_evidence` 已加 `extra_queries` 参数（双路 dense 检索按 cid 合并）；待接入 `retriever_node` 的 LLM 译写。
- eval 侧：冻结 120 题英译（`query_translations_en.json`，一次性 LLM）保持确定性，`XLING=1` 开启实测。

**实测（确定性，冻结集）**：

| 指标 | 基线 | 跨语言双路 | 变化 |
|---|---|---|---|
| **rerank Recall@6** | 0.858 | **0.892** | **+3.4pp** |
| Recall@40 上限 | 0.892 | 0.917 | +2.5pp |
| MRR / NDCG@6 | 0.707 / 0.746 | 0.735 / 0.775 | +2.8 / +2.9 |
| 向量硬漏检 | 13 | **7** | −6 |

**关键验证**：救回的 6 条恰好是「具体问题→英文 gold」的真跨语言案例（schema 查询、嵌入测试、幻觉训练机制、固态电解质、微服务扩展、数据保护条款）；**残留 7 条硬漏检几乎全是宽泛/低质题**（「涉及哪个编程语言」「关键技术概念有哪些」「网站指南分类」「学习方式有哪两种」），即次因 P2 territory。主因被确认并压下，残留回落到测量假象。
**代价**：检索 2 路（2× embed + 2× search）+ 生产侧每 query 一次 LLM 译写；reranker 候选池增大到 ≤80（≈2× rerank）。研究系统可接受。

### 跨语言方案选型：源头解决 vs 查询翻译（待评估）

P-XLing 当前用 **LLM 查询翻译**（中文 query→英文，双路检索），有效（+3.4pp）但**每个 query 多一次 LLM 调用**，推高生产 LLM 并发闸门压力。更根本的方向是让 embedding 本身跨语言对齐、省掉翻译。评估：

| 方案 | 思路 | 主 LLM 负载 | 评价 |
|---|---|---|---|
| **A. 换更强多语言 embedding** | 单 query 直接对齐英文 gold，免翻译 | **零** | 最干净，且天然支持多语言。但 bge-m3 已是开源 SOTA，未必有模型在 zh→en 上更好——须 A/B 实测。冻结集可确定性比较（gold 与 embedding 无关，无需重建）。候选：gte-multilingual-base（~600MB，小快）、multilingual-e5-large（需 query:/passage: 前缀）、API（text-embedding-3-large / cohere-multilingual，已有 `api` provider 路径） |
| **B. 索引期翻译文档** | 英文块译中文一并入库 | **更高** | ❌ 本系统语料是**每次研究新抓取的动态语料**，索引期翻译=每轮翻上千块 ≫ 翻单 query，对本架构更重，否决 |
| **C. 本地 MT 做翻译 ✅采纳** | 保留双路，翻译用本地小 MT（opus-mt-zh-en），不走主 LLM 池 | **主池零** | **已实测：opus-mt 守住 0.892（= LLM 翻译），20ms/题、77M/~300MB**，把翻译移出主 LLM 闸门、零检索损失。代价：+一个轻量本地模型 |
| **D. bge-m3 ColBERT 多向量** | 同模型 token 级跨语言 MaxSim | 零 | 同模型、无翻译；但多向量存储/算力重、Qdrant 复杂度高、收益不确定 |

> **关键架构洞察**：本系统是**动态语料**（每研究一次现抓现建索引），故「翻译 query」本就是 LLM 最省的路径（翻 1 个 query ≪ 翻整批新抓的块）。真正要消除的是「翻译挤占主 LLM 池」，而非「翻译」本身。

**决策（已定）**：放弃 A（不再追 embedding 性能），采纳 **C — opus-mt-zh-en 本地翻译**。B 否决，D 不需要。

#### C 实测（opus-mt-zh-en，确定性冻结集）

| 翻译器 | Recall@6 | Recall@40 | MRR | 延迟 | 体量 | 主 LLM 负载 |
|---|---|---|---|---|---|---|
| 无（纯 dense 基线） | 0.858 | 0.892 | 0.707 | — | — | — |
| LLM（mimo）翻译 | 0.892 | 0.917 | 0.735 | 秒级/题 | — | 每题 1 次 |
| **opus-mt 本地翻译** | **0.892** | 0.917 | 0.738 | **20ms/题(GPU)** | 77M | **零** |

opus-mt 三项要求（轻量/快/准）全达标，Recall@6 与重型 LLM 翻译**完全持平**；残留 7 条硬漏检不变（均为单 gold 假象，与翻译质量无关）。**NLLB 不必测**：opus-mt 已触及 embedding+reranker 决定的天花板，更大的 MT 救不了假象题。
依赖：`sentencepiece`（MarianTokenizer 需要，~1MB）。

### P2 — 黄金集多 Gold 标注（量化假象，P-XLing 后按需）
量化主因里有多少是「同义中文/其他英文块也可答、只标一个 gold」的假象。**先看 P-XLing 效果再决定是否上 P2**：若 P-XLing 已把漏检大幅压下、剩余假象规模小，则不必。
